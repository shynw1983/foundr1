# Foundr1 OS 訂貨システム運用設計メモ

## 目的

Foundr1 OS の訂貨システムは、複数店舗・複数ブランドの発注依頼、購入実行、納品、店舗確認、レシート確認、商品マスタ、発注先、現場記録、商品比較を一元管理するバックオフィスモジュールです。

店舗が必要なものを依頼し、購入担当が実際に買い、店舗が差異や納品を確認できる状態を作ることが主目的です。

## 用語

- `発注`: 店舗側の依頼・注文。何が必要かを表す。
- `購入`: 購入担当が実際に買う行為。数量、価格、購入先を記録する。
- `納品`: 商品が店舗へ届く、または配送される状態。
- `店舗確認`: 店舗が数量差異、価格異常、購入不可、購入メモ、納品を確認すること。
- `レシート`: 購入証跡。商品単位ではなく発注先単位でアップロードする。

「発注」と「購入」は混同しない。画面、ボタン、通知、ドキュメントで同じ意味には同じ言葉を使う。

## 画面ごとの役割

- ダッシュボード: 未処理、要確認、価格傾向、最近の状況を確認する。
- 発注依頼: 店舗側が依頼を作成し、異常や納品を確認する。
- 購入管理: 購入担当が購入、数量入力、価格入力、異常報告、配送、到着、レシートアップロードを行う。
- 発注履歴: 履歴、使用量ランキング、明細一覧、レシート確認、期間集計を行う。
- 商品マスタ: 商品、規格、参考価格、規格単価、画像、発注先、購入リンク、日本語メモを管理する。
- 発注先: 住所、連絡先、URL、チャネル種別、信頼度を管理する。
- 店舗/ブランド: 店舗、ブランド、適用関係を管理する。
- スタッフ: 役割、権限スコープ、最終ログイン、Lark 連携を管理する。
- 連絡/報告: 店舗確認や業務連絡の履歴を確認する。
- 現場記録: 新商品、発注先候補、価格情報、現場アイデアを記録する。
- 商品比較: 現行品と候補品の単位コストを比較する。

## 標準フロー

1. 店舗スタッフが発注依頼画面で商品を選ぶ。
2. 依頼担当は現在ログイン中のユーザーを初期値にする。
3. 購入担当は店舗 owner を優先し、次に owner/manager を候補にする。
4. 発注依頼を送信すると、入力中の内容はクリアされる。
5. 購入担当にアプリ内通知を送る。Lark 設定があれば Lark にも通知する。
6. 購入担当は購入管理で実数、購入価格、購入先、異常を記録する。
7. 実数を変更しただけでは店舗連絡を作らない。
8. 配送、到着、購入完了など実行操作が完了した段階で、必要な店舗確認を作る。
9. 実店舗購入は配送バッチにまとめられる。
10. ネットショップと卸業者は発注先ごとに「発注済み」「到着予定日」「到着済み」を管理する。
11. 店舗は発注依頼画面で数量差異、価格異常、購入不可、購入メモ、納品を確認する。
12. Owner は発注履歴でレシートを確認・ダウンロードし、必要に応じて履歴を削除できる。

## ステータス

### 明細

- `未購入`
- `購入済み`
- `購入不可`
- `配送中`
- `納品済み`
- `店舗確認済み`

### 依頼全体

- `未処理`
- `一部購入済み`
- `購入完了`
- `配送中`
- `納品済み`
- `店舗確認済み`

完了済みで操作できないボタンは黒背景・白文字に統一する。操作可能な主要ボタンは緑を基本にする。

## 異常と店舗確認

主な確認対象：

- 数量差異
- 価格異常
- 購入不可
- 購入メモ
- 規格違い
- 欠品
- 代替購入

重要なルール：

- 実数変更だけで連絡/報告を作らない。
- 配送や到着など、作業が確定したタイミングで店舗確認を発生させる。
- 価格異常は「基準価格を更新」または「確認して非表示」のどちらかを選べる。
- 購入不可の商品があっても、依頼全体は完了できる。
- 購入メモと購入不可の内容は店舗確認に表示する。

## レシート

レシートは発注先単位で扱う。

- 1 回の発注先購入に 1 枚アップロードする。
- 一部だけ購入し、後日残りを購入した場合は、その追加分で別のレシートをアップロードできる。
- スマートフォン写真はアップロード前に圧縮する。
- 購入済み商品があるのにレシートがない場合は「レシート未アップロード」と表示する。
- 発注履歴のレシート確認では、モーダルでプレビューし、確認済みにし、ダウンロードできる。

## 権限

- `owner`: 全権限。履歴、明細、連絡/報告の削除、マスタ管理、スタッフ管理、レシート確認ができる。
- `manager`: 広い運用管理権限。
- `buyer`: 購入管理を行う担当。
- `store_owner`: 店舗または加盟店 owner。店舗の依頼と確認、商品マスタ閲覧が中心。
- `staff`: 店舗スタッフ。発注依頼と店舗確認が中心。

権限は `employees.role` と `employee_scopes` の組み合わせで決まる。メニューは権限に合わせて表示し、権限のない項目は最初から出さない。

## 通知

アプリ内通知は `os_notifications` に保存する。

通知対象：

- 発注依頼送信時の購入担当通知。
- 配送到着後の店舗確認通知。
- ネットショップ/卸業者の到着後の店舗確認通知。
- 異常や連絡/報告の確認通知。

Lark は任意機能です。

- 社内ユーザーはメールから open_id を取得できる。
- 購入担当個人へ Lark 通知できる。
- 外部加盟店 owner は会社の連絡先範囲外になる可能性があるため、将来的には店舗ごとの Lark グループ webhook/chat_id が望ましい。

## 商品マスタ

商品マスタは、購入と比較の基礎データです。

主な項目：

- 商品名
- 日本語メモ
- 大分類 / 小分類
- 単位
- 数量 / 数量単位
- 規格
- 参考価格
- 規格単価
- 保管
- 原産国
- 適用ブランド
- メイン発注先 / 予備発注先
- 購入リンク
- 商品画像

規格単価の考え方：

- 数量がある場合は数量を優先する。
- 数量がない場合は重量を使う。
- g と kg は換算する。
- ml と L は重量換算しない。
- 例: 1 箱 200 個、1 個 1500 ml の容器は、1500 ml ではなく 200 個で単価を計算する。

商品カード右側の基本情報表示はユーザーごとに選択できる。選択内容は `employees.ui_preferences` に保存する。

## 発注先

発注先には実店舗、チェーン店、ネットショップ、卸業者、プラットフォーム店舗がある。

管理項目：

- 発注先名
- チャネル種別
- 住所
- 電話
- 担当者
- 営業時間
- 注文 URL
- 備考
- 信頼度

ネットショップと卸業者は、実店舗購入ではなく「注文して到着を待つ」フローとして扱う。

## 発注履歴とレポート

発注履歴では、次の確認ができる。

- 依頼番号単位の履歴
- 商品使用量ランキング
- 明細一覧
- レシート確認
- 店舗別集計
- 全店舗集計
- 月単位と任意期間の集計

Owner は必要に応じて履歴、明細、連絡/報告を削除できる。

## 現場記録

現場記録は、全員が参加できる提案・発見メモです。

- 新商品アイデア
- 新しい発注先候補
- 価格が安い商品
- 写真付きメモ
- コメント
- 状態管理

## 商品比較

商品比較は、現行品と候補品を単位コストで比較する機能です。

- 現行品は分類から選択する。
- 候補品は商品名、候補発注先名、購入リンク、原産国、価格、規格数量、単位を入力する。
- 国内商品は JPY を前提にする。
- 海外輸入品は外貨金額と通貨を入力する。現在は CNY を想定する。
- 為替レートで JPY 換算する。
- 運賃は kg 単価と輸入総重量から計算する。
- 商品単位が箱の場合、輸入比較では箱重量が必要になる。
- g/kg は換算するが、ml/L/個は重量換算しない。
- 比較履歴は編集、コピーして再比較、アーカイブ、復元、削除ができる。
