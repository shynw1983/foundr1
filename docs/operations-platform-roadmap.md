# FOUNDR1 品牌门店运营平台规划

## 定位

Foundr1 OS 是面向餐饮店的全系列解决方案后台。`foundr1.jp` 暂时作为前台官网域名保留，管理后台入口统一为 `foundr1.jp/os`，店铺现场操作入口使用同项目下的 `/store`。

采购系统回答：

- 门店需要什么。
- 谁去购买。
- 实际买到了什么。
- 什么时候纳品。
- 是否有レシート和异常。

未来扩展的门店运营系统回答：

- 门店应该怎么做。
- 员工有没有按标准做。
- 哪些商品、配方、手顺、检查和培训互相关联。
- 总部如何看见各门店的执行质量、成本和风险。

## 路由和产品边界

推荐保持同一个 Next.js 项目和同一个数据库，但把业务模块边界分清楚。

- `/os`: Foundr1 OS 管理后台入口。登录后根据用户权限显示可进入的管理模块图标。
- `/os/orders`: 订货系统入口，包含发注、采购、纳品、履历、レシート、发注先等流程。
- `/os/menus`: 菜单管理入口，集中管理品牌网站、POS 和电子手顺书共享的售卖菜单、规格、温度、辣度、加料等选项。
- `/os/procedures`: 店舗運営模块下的电子手顺书管理后台，用于创建、编辑、发布、归档、版本管理和适用门店设置。
- `/os/timecard`: Timecard 管理后台，后续接入出退勤规则、休息、排班、勤怠确认和集计。
- `/os/pos`: POS 管理后台，后续接入菜单设置、销售确认、会计设置、权限和売上レポート。
- `/os/sales`: 统一销售订单和经营报表后台，统计 POS、Web 预约、配送平台、手动录入等所有渠道。
- `/store`: 店铺现场工作台入口，面向店员当天操作。
- `/store/orders`: 店铺订单工作台，面向店员接单、制作、出品和完成，不限定订单来源。
- `/store/procedures`: 店铺现场电子手顺书阅读和执行入口。
- `/store/timecard`: 店员出退勤、休息、シフト确认等现场操作入口。
- `/store/pos`: 店员会计、销售、菜单操作等现场操作入口。

Foundr1 OS 的长期一级业务系统优先稳定为：`発注管理`、`店舗運営`、`タイムカード`、`POS`、`共有データ`。电子手顺书是 `店舗運営` 下的核心功能，不单独作为顶级系统；未来检查表、清洁点检、温度记录、交接班、店铺日报、稽核和培训都应继续收进 `店舗運営`。这些系统共享商品总表、门店、品牌、员工、权限、通知和操作日志等基础数据；各模块只保存自己的业务数据。`/os` 和 `/store` 不是两套系统，而是同一套数据的两种入口：`/os` 面向管理、设置、确认和分析，`/store` 面向店铺现场的快速操作。

菜单管理是品牌网站、POS 和手顺书之间的菜单源头。短期可以先从现有品牌网站导入一次，之后在 OS 里维护；长期品牌网站通过公开菜单 API 读取 OS 数据，以便菜单增减、规格变化、门店限定商品和手顺书条件保持同步。

## 订单、POS 和销售报表定位

POS、Web 预约和配送平台不是三套后台，而是三种订单入口。

最终订单流应统一为：

```text
POS / Web 予約 / Uber Eats・Wolt など
  -> 统一销售订单
  -> /store/orders 店铺制作和交付
  -> /os/sales 经营报表
```

### POS

POS 是店内现场交易终端，主要回答：

- 店员现场如何为来店客人点单。
- 如何收款、折扣、退单、取消、打印レシート。
- 如何把现场订单送入制作队列。
- 如何做班次结算和支付方式记录。

POS 的现场操作入口属于 `/store/pos`，后台设置、权限、菜单、报表属于 `/os/pos` 和 `/os/sales`。

### Web 预约和外部配送

Web 预约是线上订单入口，主要回答：

- 顾客如何提前下单、支付和查看制作状态。
- 店铺如何接收、制作和交付线上订单。

Uber Eats、Wolt 等配送平台以后也应被视为订单来源，不应单独做一套报表模型。它们进入系统后同样进入统一销售订单，再进入 `/store/orders`。

### /store/orders

`/store/orders` 的最终定位是店铺订单工作台，不是单纯的 Web 预约后台。

它应该显示所有需要店员处理的订单：

- Web 预约。
- POS 现场订单。
- Uber Eats / Wolt 等配送订单。
- 必要时的手动订单。

它只保留适合现场的能力：

- 接单。
- 制作開始。
- 受け取り可 / 出品。
- 受け渡し完了。
- 取消权限按角色控制。
- 当日简易状态和必要统计。

详细经营分析不放在 `/store/orders`，而放在 `/os/sales`。

### /os/sales

`/os/sales` 是统一经营报表，不按订单入口拆分页面。

第一版应包含：

- 日期范围。
- 品牌、门店筛选。
- 渠道筛选：`pos`、`web_reservation`、`delivery`、`manual`。
- 销售额。
- 订单数。
- 客单价。
- 取消和支付失败。
- 商品排行。
- 店铺 / 渠道对比。
- 订单明细。
- 后续 CSV 导出。

### 统一销售订单模型

当前 `store_customer_orders` 是 Web 预约先行阶段的过渡表。POS 开发前应抽象为统一销售订单模型，例如：

- `sales_orders`
  - `brand_id`
  - `store_id`
  - `channel`: `pos`、`web_reservation`、`delivery`、`manual`
  - `source_platform`: `foundr1_pos`、`nanacha_web`、`uber_eats`、`wolt` 等
  - `source_order_id`: 渠道侧原始订单 ID，POS 离线订单使用本地生成 ID
  - `terminal_id`: POS 终端 ID，非 POS 渠道可为空
  - `sync_status`: `online`、`offline_pending`、`syncing`、`synced`、`sync_failed`
  - `order_no` / `pickup_code`
  - `status`
  - `payment_status`
  - `ordered_at`
  - `source_ordered_at`: POS 终端或外部平台记录的原始下单时间
  - `synced_at`
  - `paid_at`
  - `completed_at`
  - `cancelled_at`
  - `subtotal`
  - `discount`
  - `tax`
  - `service_fee`
  - `delivery_fee`
  - `total`
  - `currency`
- `sales_order_items`
  - `sales_order_id`
  - `menu_catalog_item_id`
  - 商品名和分类快照
  - `quantity`
  - `unit_price`
  - `option_total`
  - `line_total`
  - `modifiers_json`

以后 POS、Web 预约和配送订单都写入这套模型，报表自然按渠道聚合，不再为每个入口单独做一套统计。

POS 离线订单同步到云端时，必须按 `store_id + terminal_id + source_order_id` 做幂等写入，重复上传不能生成重复销售记录。离线期间发生的交易是店内事实记录，恢复同步时即使发现菜单价格、库存或会员状态已经变化，也不能直接拒绝订单入账；应保留交易当时的本地快照，并额外产生价格差异、库存异常或会员积分待处理状态。

## 近期优先级

1. 电子手顺书。
2. 开店、闭店、清洁等检查表。
3. 配方 / BOM 成本管理。
4. 库存与损耗。
5. 员工培训与确认记录。
6. 门店稽核。
7. 经营分析看板。
8. 加盟 / 多品牌后台。
9. AI、OCR、自动建议。

优先打通的闭环：

```text
商品总表
  -> 发注 / 采购
  -> 纳品 / レシート
  -> 配方 / 手顺书
  -> 门店执行
  -> 检查 / 培训 / 成本分析
```

## 连锁经营负责人视角的近期路线

截至 2026-06-07，Foundr1 OS 已经不是单一采购后台，而是接近餐饮连锁 OS 雏形的系统。接下来不应继续优先横向堆新模块，而要先把销售、原価、库存、执行、人员五条线闭成经营闭环。否则模块越多，数据口径越容易散，总部每天仍然无法依赖系统判断门店真实状态。

负责人提醒：

- 先闭环，再扩模块。
- 先统一经营口径，再做漂亮看板。
- 先让总部每天能发现异常，再补低频功能。
- 先把 POS / Web 预约 / 厨房 / 会员 / 分析跑成一条线，再接更多外部渠道。
- 每新增一个功能，都要问它是否能改善売上、原価、库存、执行质量、人员训练或风险控制。

### 1. 先统一销售事实表

当前 POS 和 Web 预约主要先写入 `store_customer_orders`，再同步到 `sales_orders`；销售分析主要读取 `sales_orders`。上线前必须把 `sales_orders` / `sales_order_items` 明确为统一销售事实表，而不是松散的报表同步副本。

必须完成：

- POS、Web 预约、外卖平台、手动订单都稳定进入同一销售订单模型。
- 取消、返金、部分返金、优惠券、积分、レシート、厨房状态都能追溯到同一订单。
- 增加一致性检查：`store_customer_orders` 与 `sales_orders` 的订单数、金额、状态是否一致。
- `/os/sales`、`/os/analytics/sales`、`/os/analytics/profit` 使用同一经营口径。
- 后续逐步减少只面向 Web 预约的过渡逻辑。

### 2. 打通销售到原価和粗利

销售额没有原価，只能看营业热闹，不能真正管理门店。下一步应优先把菜单、配方、商品总表和采购价格接起来。

必须完成：

- 菜单商品绑定配方 / BOM。
- BOM 绑定商品总表 `products`。
- 商品成本优先取最近采购价，其次取基准价或供应商价。
- 每个销售订单 item 能计算理论原価。
- 报表按品牌、门店、商品、渠道显示売上、原価、粗利、粗利率。
- 食材、包材、消耗品进入商品原価；房租、水电、通信等继续作为月次経費，不混入口径。
- maamaa 这类 buildable 商品按实际加料、重量、规格和选项计算成本，不按固定 SKU 粗略计算。

### 3. 补库存、损耗和盘点

采购和销售之间缺少库存层，就无法解释理论成本和实际成本的差异。库存模块不要一开始做复杂仓储系统，先做餐饮门店可落地的最小闭环。

第一版闭环：

```text
納品入库
  -> 销售理论扣减
  -> 废弃 / 损耗记录
  -> 月末盘点
  -> 理论库存和实际库存差异
```

必须完成：

- 入库记录来自納品 / 采购。
- 出库记录来自销售理论消耗、手动领用、调拨。
- 损耗记录覆盖废弃、过期、制作失败、试吃、员工餐。
- 支持日盘、周盘、月盘。
- 显示理论库存 vs 实际库存。
- 低库存、过期、异常损耗触发提醒。
- 店铺现场入口放在 `/store`，保持快速录入；管理和分析放在 `/os`。

### 4. 店舗運営从手順書扩展到执行记录

电子手顺书是标准，但连锁总部真正要看的是门店有没有按标准执行。`店舗運営` 不应只停在 SOP 阅读。

必须完成：

- 开店检查表。
- 闭店检查表。
- 清洁检查表。
- 温度记录。
- 设备点检。
- 交接班记录。
- 异常报告。
- 总部下发任务。
- 逾期 / 未执行提醒。
- 门店执行率看板。

原则：现场入口在 `/store`，管理、模板、稽核和分析在 `/os`；不要把电子手顺书、检查表、培训、稽核拆成互不相干的系统。

### 5. 培训和岗位认证闭环

Timecard 和员工档案已经具备基础，但总部还不能判断员工是否会按标准工作。培训记录是连锁复制能力的一部分。

必须完成：

- 手順書已读确认。
- 新员工入职培训路径。
- 岗位技能等级，例如收银、厨房、开店、闭店、店长。
- 新品上线培训确认。
- 简单测验 / 考试。
- 店长查看员工训练进度和未完成项。
- 员工离职、调店、权限变更时保留培训记录。

### 6. POS 上线前的经营控制

POS 不只是收银页面，而是现金、折扣、返金和会员权益的风险入口。正式上线前必须压实控制点。

必须完成：

- 日次レジ締め形成闭环。
- 现金差异必须填写理由。
- 取消 / 返金权限严格按角色控制。
- 折扣有权限、原因和审计记录。
- 会员优惠券防重复使用。
- POS 订单稳定进入厨房、取餐屏、销售分析和会员积分。
- 外部支付终端失败、现金差异、断网补录都要有可追踪状态。
- 营业日和自然日口径明确，尤其处理跨午夜营业。

POS 还必须具备断网和外部服务故障时的最低营业能力。来店客人的基础交易不能因为 Neon、Blob、Lark、Square、KOMOJU、通知、分析或其他外部服务故障而整体停止。

离线 / 降级模式必须完成：

- `/store/pos` 支持本地营业数据快照，包括菜单、价格、税率、门店设置、终端设置、レシート模板和基础权限。
- POS 页面和关键静态资源可被 Service Worker 缓存；终端本地用 IndexedDB 保存快照、未同步订单和同步队列。
- 断网时允许现金交易继续完成，生成本地订单号，打印或显示レシート，并把订单标记为 `offline_pending`。
- 外部支付服务不可用时不能伪造已支付状态；必须清楚降级为现金、稍后处理或店铺允许的其他安全流程。
- 网络恢复后自动同步离线订单，后端按 `store_id + terminal_id + source_order_id` 幂等写入。
- 同步失败、重复订单、价格差异、库存异常、会员积分延迟入账都要进入可查看的异常状态。
- Lark、通知、图片上传、分析写入、会员查询和厨房屏推送失败时，不得阻止 POS 基础收银；这些能力应进入待重试或降级提示。
- POS 端明确显示 `オンライン`、`オフライン`、`同期中`、`同期失敗` 等状态，避免店员误判交易是否已同步。
- 如果终端当天从未成功下载过营业数据快照，可以进入受限模式；如果已有有效快照，应允许继续营业并在恢复网络后补同步。

### 7. 会员体系要能指导复购

会员、积分和优惠券已经有基础，但还需要从“能发券”变成“能评价运营效果”。

必须完成：

- 会员分层：新客、回头客、沉睡客、高频客。
- 生日券、沉睡唤醒券、注册券等都有发放、使用和效果统计。
- POS 会员和 Web 预约会员身份合并。
- 优惠券使用进入销售、毛利和结算统计。
- maamaa / nanacha 的会员订单详情、キャンセル、返金规则保持一致。
- 后台能看到券发了多少、用了多少、带来多少销售、损失多少毛利。

### 8. 外卖和导入渠道标准化

Uber Eats、Wolt、手动导入、Square、KOMOJU 等都应映射到统一销售模型，不应每个平台形成独立报表。

必须完成：

- 统一 channel：`pos`、`web_reservation`、`delivery`、`manual`。
- 统一 source platform：`foundr1_pos`、`nanacha_web`、`maamaa_web`、`uber_eats`、`wolt` 等。
- 处理导入失败、重复订单、平台手续费、平台补贴、取消单。
- 区分销售额、平台扣费、实际入金。
- 后续支持渠道利润分析。

### 9. 总部异常看板

`/os` 首页应优先显示异常，而不是只展示模块入口。老板每天要先知道哪里出问题。

第一版异常项：

- 今日未处理订单。
- 厨房超时订单。
- 未締め POS 班次。
- 现金差异。
- レシート未上传。
- 発注未納品。
- 欠品 / 售罄商品。
- 打卡异常。
- 检查表未完成。
- 高损耗商品。
- 销售低于预期门店。
- 投诉 / 反馈未处理。

### 10. 文档和产品命名收口

项目已从早期采购后台演进为 Foundr1 OS，文档、角色名和路由表述需要同步清理。

必须完成：

- README 更新为当前 `/os` / `/store` 架构。
- 明确旧 `buyer` 角色是否废弃。
- `PROJECT_CONTEXT.md` 的未确认信息重新核对。
- docs 中 roadmap 和实际页面状态对齐。
- 建立上线检查表。

### 建议执行顺序

第一阶段，2-3 周：

1. 销售事实表统一和一致性检查。
2. POS 离线收银、服务故障降级和同步队列的最小闭环。
3. POS / Web 预约 / 厨房 / 会员 / 分析端到端验证。
4. 原価计算最小闭环：菜单 item -> BOM -> 商品采购价 -> 理论成本。
5. `/os` 首页异常看板第一版。
6. README 和产品文档收口。

第二阶段，1-2 个月：

1. 库存、废弃、盘点。
2. 开店 / 闭店 / 清洁 / 温度检查表。
3. 培训确认和岗位认证。
4. 外卖渠道导入标准化。
5. 粗利、损耗、人件费联动的店铺月报。

第三阶段：

1. 加盟 / 多法人结算。
2. 供应商价格自动异常检测。
3. OCR レシート。
4. AI 生成手順書 / 检查表草稿。
5. 店铺稽核和改善任务闭环。

## 电子手顺书

第一阶段目标是可落地的门店执行手顺书，不做复杂知识库。

基础能力：

- 品牌 / 门店可见范围。
- 饮品、备料、清洁、开店、闭店等分类。
- 步骤式内容。
- 每一步支持图片、说明、注意事项和关联商品。
- 平板横屏阅读模式。
- 移动端可读。
- 桌面端可管理和预览。
- 草稿 / 发布。
- 简单版本号。
- 从商品总表选择关联商品。

建议数据关系：

- `procedure_books`: 手顺书主体。
- `menu_sources`: 品牌菜单来源，例如官网、Lark、代码内菜单、API。
- `menu_catalog_items`: 可销售对象。nanacha 这类固定饮品是 `fixed_product`，maaamaa 这类麻辣烫基础流程是 `buildable_product`。
- `menu_option_groups` / `menu_options`: 会影响菜单和手顺的变量，例如尺寸、温度、甜度、冰量、辛さ、痺れ、配料。
- `procedure_variants.condition_json`: 手顺差异条件，例如 `{"size":"R","temperature":"ICE"}` 或 `{"heat":"hot","numb":"numb"}`。
- `procedure_steps`: 步骤。
- `procedure_step_products`: 步骤和商品总表的关联。
- `procedure_versions`: 版本。
- `procedure_store_assignments`: 适用门店。
- `procedure_media`: 图片、视频等素材。

## 设计原则

- 优先平板横向布局，同时保证移动和桌面可用。
- 门店执行页面应该直接可操作，不做营销式页面。
- 采购、手顺书、培训、稽核可以共享商品总表，但不要混淆页面职责。
- 权限继续使用 role + scope，品牌、门店、供应商和未来手顺书可见范围都应走同一套思路。
- 新增可见文本时继续维护日文、简体中文、繁体中文翻译。
- 当前系统未正式上线，新增模块可以优先选择干净的数据结构，不为早期草案保留兼容负担。

## 配置化与对外销售准备

Foundr1 自用阶段可以优先按当前门店运营方式把流程跑顺，但如果未来把 Foundr1 OS 提供给其他公司，不能长期依赖为每家公司修改代码。凡是“Foundr1 自己这样用，但其他公司可能不同”的逻辑，都应逐步抽象成系统内设置。

建议把定制需求分层：

- 系统级配置：公司名称、Logo、品牌色、默认语言、支持语言、Store 顶部时钟、用户信息显示方式、导航分组方式。
- 模块级配置：哪些模块启用，例如菜单、预约、POS、Timecard、手顺书、报表；每个模块的页面显示方式和现场操作入口。
- 业务规则配置：营业时间、预约提前时间、临时休业、堂食 / 外卖 / 预约 / POS 渠道、门店可售范围、选项 / 加料是否独立管理。
- 权限配置：总部、店长、员工、加盟商能看见和操作哪些页面、字段、按钮和门店范围。

近期开发原则：

- 可以先写出 Foundr1 自用默认值，但代码结构要预留配置入口。
- 遇到 UI 排列、模块开关、选项显示、权限差异、业务规则差异时，优先判断它是否应该成为配置，而不是直接写死。
- Store 现场界面的定制项尤其要留意，例如用户信息显示为头像还是姓名、销售状态里选项是否独立分类、是否允许店员改价格或只允许改售罄状态。
- 稳定后新增 `/os/settings` 或 `/os/company-settings`，集中管理公司级、品牌级、门店级、模块级配置。

### Store 销售状态配置

Store 的销售状态页面不应长期写死为“商品 + 选项 / 加料”。不同客户可能需要门店现场控制的对象不同，因此销售状态管理对象应配置化。

可配置对象示例：

- 商品。
- 选项 / 加料。
- 尺寸、温度、甜度、辣度、麻度等规格项。
- 套餐组件。
- 未来 POS、Web 预约、外卖平台等渠道可售状态。

可配置显示方式示例：

- 只显示商品。
- 商品和选项 / 加料混合显示。
- 按对象类型独立分类显示，例如 Foundr1 当前默认的“商品分类 + オプション・トッピング”。
- 按客户自定义业务分组显示。

可配置权限示例：

- 门店只能切换售罄 / 恢复。
- 门店可以填写现场备注。
- 门店是否允许改价格。
- 门店是否允许控制 Web、POS、外卖渠道的显示 / 停止。

Foundr1 当前默认值：

- 商品可由门店控制。
- 选项 / 加料可由门店控制。
- 选项 / 加料在 Store 左侧作为独立分类显示。
- 门店只改销售状态和现场备注，不直接改菜单主数据。
