# FOUNDR1 品牌门店运营平台规划

## 定位

Foundr1 OS 是面向餐饮店的全系列解决方案后台。`foundr1.jp` 暂时作为前台官网域名保留，后台入口统一为 `foundr1.jp/os`。

采购系统回答：

- 门店需要什么。
- 谁去购买。
- 实际买到了什么。
- 什么时候纳品。
- 是否有レシート和异常。

未来扩展的门店运营系统回答：

- 门店应该怎么做。
- 员工有没有按标准做。
- 哪些商品、配方、手顺、检查和培训互相关联。
- 总部如何看见各门店的执行质量、成本和风险。

## 路由和产品边界

推荐保持同一个 Next.js 项目和同一个数据库，但把业务模块边界分清楚。

- `/os`: Foundr1 OS 后台入口。登录后根据用户权限显示可进入的模块图标。
- `/os/orders`: 订货系统入口，包含发注、采购、纳品、履历、レシート、发注先等流程。
- `/os/procedures`: 电子手顺书模块，用于创建、编辑、发布、归档、版本管理、适用门店设置和现场阅读。
- `/os/timecard`: Timecard 打卡模块，后续接入出退勤、休息、排班和勤怠确认。
- `/os/pos`: POS 模块，后续接入销售、会计、菜单和売上レポート。

订货、电子手顺书、Timecard、POS 都是 Foundr1 OS 下的平级业务模块。它们共享商品总表、门店、品牌、员工、权限、通知和操作日志等基础数据；各模块只保存自己的业务数据。

## 近期优先级

1. 电子手顺书。
2. 开店、闭店、清洁等检查表。
3. 配方 / BOM 成本管理。
4. 库存与损耗。
5. 员工培训与确认记录。
6. 门店稽核。
7. 经营分析看板。
8. 加盟 / 多品牌后台。
9. AI、OCR、自动建议。

优先打通的闭环：

```text
商品总表
  -> 发注 / 采购
  -> 纳品 / レシート
  -> 配方 / 手顺书
  -> 门店执行
  -> 检查 / 培训 / 成本分析
```

## 电子手顺书

第一阶段目标是可落地的门店执行手顺书，不做复杂知识库。

基础能力：

- 品牌 / 门店可见范围。
- 饮品、备料、清洁、开店、闭店等分类。
- 步骤式内容。
- 每一步支持图片、说明、注意事项和关联商品。
- 平板横屏阅读模式。
- 移动端可读。
- 桌面端可管理和预览。
- 草稿 / 发布。
- 简单版本号。
- 从商品总表选择关联商品。

建议数据关系：

- `procedure_books`: 手顺书主体。
- `procedure_steps`: 步骤。
- `procedure_step_products`: 步骤和商品总表的关联。
- `procedure_versions`: 版本。
- `procedure_store_assignments`: 适用门店。
- `procedure_media`: 图片、视频等素材。

## 设计原则

- 优先平板横向布局，同时保证移动和桌面可用。
- 门店执行页面应该直接可操作，不做营销式页面。
- 采购、手顺书、培训、稽核可以共享商品总表，但不要混淆页面职责。
- 权限继续使用 role + scope，品牌、门店、供应商和未来手顺书可见范围都应走同一套思路。
- 新增可见文本时继续维护日文、简体中文、繁体中文翻译。
- 当前系统未正式上线，新增模块可以优先选择干净的数据结构，不为早期草案保留兼容负担。
