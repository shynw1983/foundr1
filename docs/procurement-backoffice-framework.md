# Foundr1 OS 订货系统业务说明

## 项目定位

Foundr1 OS 的订货系统模块是面向连锁餐饮、加盟店和轻人力门店运营的发注与采购协同后台。它解决的是“门店需要什么”“谁去买”“实际买到了什么”“是否需要门店确认”“价格和供应商是否值得调整”这些日常问题。

系统目前不做招商前台，而是优先做内部运营后台。未来可以继续扩展为加盟店管理、成本分析、供应商协同和商业化 demo。

## 核心原则

### 用语原则

- `发注 / 発注`：门店发起的需求、订单、依赖。
- `购买 / 購入`：采购担当实际执行购买。
- `纳品 / 納品`：商品到达门店或交付给门店。
- `门店确认 / 店舗確認`：门店对数量差异、价格异常、购买不可、采购备注、到货情况进行最终确认。
- `レシート`：日本现场使用的收据名称，不使用“小票”。

同一个动作不能在不同页面用不同词表达。下订单、发注依赖、发注履历属于发注侧；采购管理、购买价格、购买数量属于购买侧。

### 工作边界

- 发注依赖界面：门店和门店相关人员使用，负责下单、查看状态、确认异常。
- 采购管理界面：采购人员使用，负责购买、记录实际数量、价格、异常、配送和到货。
- 订单履历界面：管理和复盘使用，负责统计、删除历史、确认/下载レシート。
- 商品总表：管理商品、供应商、规格、价格、适用品牌、购买链接和日本语备注。

## 标准流程

1. 门店用户在发注依赖界面选择门店、品牌和商品。
2. 需求担当默认当前登录用户，并且当前用户在候选人中优先级最高。
3. 采购担当默认店铺 owner，其次 owner/manager，未来可替换为专门采购担当。
4. 用户提交发注依赖后，页面清空当前下单内容，避免误以为还未提交。
5. 系统生成站内通知；配置 Lark 时，同时通知采购担当个人。
6. 采购担当进入采购管理界面处理商品。
7. 采购担当记录实际数量、实际购买价格、购买渠道和异常报告。
8. 只有当采购执行动作完成后，系统才把数量差异、价格异常、不可购买、采购备注转成门店需要确认的信息。
9. 实体店采购可以把已购买商品组成配送批次，并进入配送/到货/门店确认流程。
10. 网店和批发商按供应商维度记录已下单、预计到达日、到货状态。一个订单内多个网店可以有不同到货时间。
11. 发注先维度上传レシート。一次只买了一部分时，后续购买可以上传新的发注先履行记录对应的レシート。
12. 门店在发注依赖界面完成确认。
13. Owner 在订单履历中查看统计、レシート确认、下载、删除历史记录。

## 状态逻辑

### 明细状态

- `未購入`：还没有购买。
- `購入済み`：已购买。
- `購入不可`：本次无法购买，但订单整体仍可完成。
- `配送中`：已进入配送。
- `納品済み`：已到货或已交付。
- `店舗確認済み`：门店确认完成。

### 订单层状态

- `未処理`：尚未开始采购。
- `一部購入済み`：部分商品已购买。
- `購入完了`：全部可处理商品完成购买或标记为不可购买。
- `配送中`：有商品正在配送。
- `納品済み`：已到货但等待门店确认。
- `店舗確認済み`：门店侧确认完成。

完成后不能再操作的按钮统一黑底白字。可操作按钮使用绿色或白底蓝字等明确可点击状态。

## 异常和确认

异常报告用于记录采购现场真实情况。

当前异常类型包括：

- 数量差异
- 价格异常
- 不可购买
- 采购备注
- 规格不一致
- 缺品
- 替代购买

工作流约束：

- 改变实数本身不应立刻产生门店联络。
- 点击配送、到货、购买完成等执行动作后，才生成需要门店确认的信息。
- 价格异常可以有两个处理方式：更新基准价格，或确认后隐藏。
- 不可购买不阻止订单整体完成。
- 采购备注和不可购买内容必须出现在门店确认信息里。

## レシート / 收据

レシート按发注先维度管理，而不是按商品维度管理。

设计规则：

- 一家供应商一次采购上传一张レシート。
- 如果同一家供应商只采购了一部分，后续再次采购时可以再上传一张。
- 上传照片时前端应自动压缩，避免手机照片超过上传限制。
- 采购管理中如果已购买但未上传，需要明确显示“レシート未アップロード”。
- 订单履历中有レシート确认板块，Owner 可以预览、确认、下载。
- 预览使用浮窗，不新开标签页。

## 权限设计

角色和范围共同决定权限。

- `owner`：全权限。可以删除订单履历、订单明细、联络报告，管理员工、门店、供应商、商品和レシート确认。
- `manager`：运营管理。可以查看大部分数据并处理管理工作。
- `buyer`：采购担当。处理采购、异常、配送、到货、レシート上传。
- `store_owner`：加盟店或门店老板。可以查看商品总表，但通常只能查看不能编辑；可以下单和确认本店相关事项。
- `staff`：门店员工。创建发注依赖、查看本店状态、处理门店确认。

权限还要结合 `employee_scopes`。菜单必须按当前登录用户权限显示，不能全量显示后让无权限按钮点了没反应。

## 通知设计

站内通知保存在 `ops_notifications`，是每个用户自己的通知。

当前通知场景：

- 发注依赖送信后通知采购担当。
- 网店/批发商到货后通知相关门店确认。
- 配送批次到货后通知相关门店确认。
- 异常或联络报告需要确认时通知相关人员。

Lark 是可选增强：

- 内部员工可通过邮箱查 open_id。
- 发注通知可以发给采购担当个人。
- 外部加盟店老板如果不在公司通讯录里，个人 open_id 会受到限制；更适合用门店群 webhook/chat_id。
- 未来需要实现门店绑定 Lark 群，发注通知同时发给采购担当个人和门店群。

## 商品总表

商品总表是运营基础数据中心。

重要字段：

- 商品名
- 日本语备注：中文商品名或进口商品给日本员工看的补充说明。
- 大分类 / 小分类
- 单位
- 数量与数量单位
- 规格
- 参考价格
- 规格单价
- 保管方式
- 原产国
- 适用品牌
- 主要发注先 / 备用发注先
- 购买链接
- 图片

规格计算规则：

- 如果填写了数量，优先用数量计算单价。
- 如果没有数量，再用重量规格计算。
- 克和千克可自动换算。
- 毫升和升不参与重量换算。
- 例如一箱 200 个、每个 1500 ml 的杯子，应按 200 个算单个成本，不能按 1500 ml 算。

用户可以选择商品卡片右侧显示哪些基本信息，选择保存在本人 `ui_preferences` 里。选择太多时必须换行或压缩布局，不能横向溢出。

## 供应商管理

供应商可以是：

- 实体店
- 连锁超市
- 网店
- 批发商
- 平台店铺

供应商资料应包括：

- 名称
- 类型
- 地址
- 电话
- 联系人
- 营业时间
- 购买 URL
- 备注
- 可信度

商品总表选择供应商时，应支持快速新增供应商的简易弹窗。

网店和批发商在流程上更接近“提交订单后等待到货”，不是“到店购买后配送”。因此它们按供应商维度记录已下单和预计到达日。

## 订单履历与报表

订单履历不只是明细列表，需要用于复盘。

当前包括：

- 订单履历
- 商品使用量排行
- 明细一览
- レシート确认
- 店铺筛选
- 全店汇总
- 时间区间选择
- 月份快速选择和自定义日期范围

Owner 可以删除订单、订单明细和联络报告。

## 现场记录

现场记录用于任何人提交现场发现：

- 新商品创意
- 供应商现场发现
- 更便宜渠道
- 需要后续讨论的商品

它是意见和素材收集板块，所以所有角色都可以参与提交。记录支持照片、批注、状态和删除权限控制。

## 商品比较

商品比较用于把现有商品与候选商品做成本对比。

设计规则：

- 现有商品按分类选择，避免所有商品堆在一张列表里。
- 候选商品有独立商品名、候选供应商名、购买链接、原产国、价格、规格数量、单位和备注。
- 国内商品默认日元。
- 海外进口品输入外币金额和币种，目前支持人民币。
- 汇率按当前实际汇率换算成日元。
- 运费按每 kg 运费乘以进口总重量计算。
- 如果商品单位是箱，进口比较时需要填写每箱重量。
- 克和千克自动换算，毫升、升、个不做重量换算。
- 历史结果支持编辑、复制后重新比较、归档、恢复和删除。

## 商业化方向

这个系统对麻辣烫、奶茶、轻餐、外带和依赖食材堆叠的行业有优势，因为这些行业成本由 SKU、规格、供应商和购买频率决定。

更适合强调的价值：

- 降低漏买和错买。
- 让兼职或加盟店也能按标准流程采购。
- 记录价格异常与供应商变化。
- 形成商品单价和历史趋势。
- 用レシート和实际购买记录支撑成本复盘。
- 给总部看各店使用量、异常率和高成本商品。

## 暂缓但已记录的事项

- 外部成员/加盟店老板 Lark webhook 群通知。等待有门店外部群后再验证。
- 波动型价格的参考价格策略和历史趋势图。蔬菜等季节波动商品可考虑“参考区间”而不是单一基准价。
- レシート OCR 自动识别。
- 专门采购担当角色细分。
- 安全库存和自动补货提醒。
