# 门店采购协同后台框架

## 项目定位

这个后台系统服务于旗下餐饮品牌的日常采购协同，优先解决多门店、多品牌场景下的采购需求提交、采购执行、异常反馈、送达确认和价格趋势维护问题。

当前品牌特性是线上与外带优先、轻人力运营、一人可管理门店现场。系统第一阶段不建设招商前台，但技术架构预留未来 Next.js 前台官网、招商页、加盟管理和多门店扩展能力。

## 核心目标

- 减少门店与采购担当之间的重复沟通。
- 避免漏买、重复买、买错规格、送错门店。
- 让缺货、价格异常、替换采购等信息可追踪。
- 维护主要食材价格，逐步形成价格趋势判断。
- 支持未来新增门店、品牌、实体店和加盟体系。

## 基础对象

### 门店

- 门店名称
- 地址
- 负责人
- 联系方式
- 营业状态
- 可经营品牌

### 品牌

- 品牌名称
- 品牌类型，例如奶茶、麻辣烫
- 适用门店
- 品牌负责人
- 运营状态

### 员工与角色

- 店员：提交采购需求、查看反馈、确认收货
- 店长：审核或补充门店需求、查看门店记录
- 采购担当：处理采购单、记录实际数量和异常
- 管理员：维护门店、品牌、商品、供应商、价格和权限

员工权限需要由“角色 + 可见范围”共同决定。采购担当也要区分总部采购担当、店铺采购担当、品牌采购担当或供应商采购担当。

权限范围建议：

- 总部采购担当：可以查看和处理全部门店、全部品牌的采购依赖。
- 店铺采购担当：只能查看和处理被分配门店的采购依赖。
- 品牌采购担当：只能查看和处理对应品牌相关的采购依赖。
- 店铺员工：只能创建所属门店采购依赖，并查看本店反馈与配送状态。
- 管理员：可以配置门店、品牌、员工、角色和权限范围。

系统在发送采购依赖时，需要根据送达门店和对应品牌自动判断可见采购担当，确保不同采购担当看到的信息隔离并分开管理。

### 商品

涵盖备品、消耗品、食材、包材和设备耗材。

基础字段：

- 商品名称
- 分类
- 适用品牌
- 主要购买商家
- 参考价格
- 默认需下单数量
- 单位
- 规格备注
- 是否关键物料
- 是否容易涨价
- 是否需要冷链
- 商品照片，后续扩展

同一种食材或备品可能被多个品牌共用，这种情况不要重复建多个商品。商品本体只建一条，再通过品牌别使用设置记录不同品牌怎么使用。

品牌别使用设置建议字段：

- 商品
- 品牌
- 用途
- 默认下单数量
- 规格备注
- 优先级
- 显示顺序
- 该品牌是否可下单

未来数据结构建议：

- `products`：商品本体
- `product_brand_usages`：商品和品牌的使用关系、用途、默认数量

### 商品与供应商关系

同一种商品可能有多个供应商或购买渠道，所以商品本体和供应商价格不能绑死在一起。

商品管理只记录商品是什么，供应商别设置记录在哪里买、多少钱、优先级如何、是否适合紧急采购。

供应商别设置建议字段：

- 商品
- 供应商
- 渠道角色：主供应商、备用供应商、紧急购买渠道
- 参考价格
- 最小购买量
- 到货时间
- 适用门店
- 适用品牌
- 缺货时是否可替代
- 备注

采购处理时，系统默认显示主供应商；如果主供应商缺货、价格异常或来不及配送，采购担当可以切换备用或紧急渠道。实际购买渠道、实际数量、实际价格和异常原因需要记录在采购明细中。

未来数据结构建议：

- `products`：商品本体
- `suppliers`：供应商本体
- `product_supplier_options`：商品与供应商的对应关系、优先级、参考价格
- `purchase_order_items`：采购依赖中的商品和数量
- `purchase_actuals`：实际购买渠道、购买数量、实际价格和异常内容

### 供应商 / 购买渠道

- 供应商名称
- 联系方式
- 平台或地址
- 主要品类
- 营业时间
- 配送能力
- 付款方式
- 稳定性评价
- 备用供应商

供应商需要拆成两层：

- 供应商本体：公司、品牌、平台或 EC 店铺名称。
- 购买据点：具体网店、实体店、连锁店分店。

购买据点建议字段：

- 所属供应商
- 据点名称
- 类型：网店、实体店、连锁店分店
- 地址或服务区域
- 营业时间
- 购买方式：配送、到店购买、到店取货、电话下单
- 对应门店
- 是否支持配送
- 是否适合紧急采购
- 备注

实际采购记录中，不只记录供应商，还要记录具体购买据点。这样以后可以知道同一个连锁店到底是哪家分店价格稳定、哪家容易缺货、哪家距离某个门店最近。

未来数据结构建议：

- `suppliers`：供应商本体
- `supplier_locations`：网店、实体店、连锁分店等购买据点
- `product_supplier_options`：商品和供应商的对应关系，必要时绑定推荐据点
- `purchase_actuals`：实际购买供应商、购买据点、数量和价格

## 采购流程

1. 门店员工根据分类、品牌和常用模板选择商品。
2. 员工填写下单数量、截止采购时间、优先级、送达门店和对应品牌。
3. 采购单进入待采购状态。
4. 采购担当按采购单执行采购。
5. 采购担当记录实际采购数量、缺货、替换商品、异常价格或规格问题。
6. 采购完成后向门店反馈。
7. 配送完成后，门店确认收货。
8. 有异常的采购单进入跟进状态，直到关闭。

## 采购单状态

- 草稿
- 已提交
- 待采购
- 采购中
- 部分采购完成
- 已采购
- 配送中
- 已送达
- 门店已确认
- 有异常
- 已关闭

## 异常类型

- 缺货
- 价格异常
- 规格不一致
- 数量不足
- 商品替换
- 配送延迟
- 商品损坏
- 门店拒收
- 临时加购
- 重复下单

每条异常建议记录商品、说明、处理方案、是否需要门店确认、是否影响营业和后续跟进时间。

## 第一阶段执行原则

第一阶段优先做成可用后台原型，不追求复杂自动化。重点是把采购流程跑顺，让门店能下单，采购担当能处理，管理员能维护商品和价格，老板能看到关键状态和趋势。
